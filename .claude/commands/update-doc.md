# update-doc

**用途**：mengshu 统一文档管理入口，按目标路由到对应技能。

mengshu 有两套文档体系，本命令负责识别意图后路由：

| 目标 | 文档体系 | 输出位置 | 承接技能 |
|------|---------|---------|---------|
| 内部过程文档 | 9 个编号目录（01~09） | `.memory-docs/original-docs/` | `document-workflow` |
| 对外开源文档 | guides/api/architecture/design | `docs/` | `documentation` |

> 单一事实来源（设计文档 D-01~D-23）两处都有：内部完整版 `.memory-docs/.../04-design/` + 开源精简版 `docs/design/`，修改后需同步。

---

## 路由规则

### 1. 内部过程文档 → document-workflow 技能

触发场景：

| 用户说 | 创建/更新文档 |
|--------|--------------|
| "初始化文档系统"、"建立文档规范" | 9 目录结构 + 配置（初始化模式） |
| "需要支持 XX 功能" | 业务需求 + 系统需求 |
| "数据库 Schema 变了" | 数据库设计文档 |
| "XX 接口报错 / 发现 Bug" | 缺陷记录 + 变更日志 |
| "准备发布 v1.x" | 变更日志 |
| "设计 XX 模块" | 模块设计（04.1）或详细设计（04.2） |
| `/doc-health`、"检查文档状态" | 健康检查（运行脚本，输出评分） |

执行：参见 [document-workflow/SKILL.md](~/.claude/skills/docs/document-workflow/SKILL.md)。

### 2. 对外开源文档 → documentation 技能

触发场景：

| 用户说 | 创建/更新文档 |
|--------|--------------|
| "写用户指南"、"快速开始" | `docs/guides/` |
| "更新 API 参考"、"CLI 命令文档" | `docs/api/` |
| "架构概览"（对外） | `docs/architecture/` |
| "核心设计文档"（对外精简版） | `docs/design/` |
| "把内部设计转成对外文档" | 对应 docs/ 子目录 + 红线过滤 |

执行：参见 [documentation 技能](../skills/documentation/SKILL.md)，遵循面向用户的红线（不暴露内部过程、实现细节、临时信息）。

### 3. 不确定目标时

先问用户：这是写给团队内部看的过程记录，还是给最终用户看的对外文档？

---

## 执行流程

```
意图识别 → 路由到 document-workflow / documentation → 判断复杂度 → [复杂时] 出计划等确认 → 执行 → 告知结果
```

**何时先出计划**：
- 初始化（创建 9 个目录 + 多文件）
- 单次涉及 3 个以上文档的更新
- 内部设计转对外文档（需红线过滤，先列出将暴露/隐藏的内容）

**何时直接执行**：
- 单文档更新（如只新增一个 API doc）
- 健康检查

---

## 关联命令

当更新涉及**模块新增或架构调整**时，建议同步执行：
- `/update-codemaps` — 更新代码结构索引（给 AI 用的代码导航地图）

---

**记住**：文档与代码同等重要。内部文档求全（过程可追溯），对外文档求精（用户视角、不泄实现）。
