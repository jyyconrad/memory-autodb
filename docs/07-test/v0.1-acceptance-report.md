## 1. 验收结论

v0.1 的核心目标——**单 appId 内可验证的 Working Context 闭环（5 slot 低延迟注入、注入安全可解释、scope 隔离正确、记忆可治理）**——在**模块与集成测试层面全部达标**：

- 全量测试：57 文件 / 401 通过 / 1 跳过（live OpenAI 测试）
- 类型检查：`npx tsc --noEmit` 零错误
- v0.1 黄金集：30/30 通过，关键记忆召回 100%（≥ 80% 门槛）
- safety 黄金集：40/40 通过，误注入率 0%
- context_fast 本地 P95：1ms（远低于 250ms / 80ms 门槛）
- 覆盖率：82.37% statements / 82.85% lines（超 80%）；69.05% branch（略低于 70%）

**一个诚实的范围说明**：候选区"运行中 observation 自动抽取"入口（observe 工具 + extract job worker）尚未接线（见 §6.1）。Console 审核闭环（列出 pending → approve → 写主库）已通过 daemon 端到端验证；候选区后端、Console API/前端、scope policy、敏感过滤、eval 基础设施均已实现并测试。

## 6. 已知缺口（v0.1 后续 / v0.2）

### 6.1 候选区运行时编排接线（部分已收尾）

**已收尾（2026-06-12）**：Console 后端已通过 `serve` 命令注入 daemon。
- `index.ts` 构造共享 `InMemoryCandidateRepository` + `CandidateReviewService`（promoteCandidate 经 `candidateToMemoryRecord` 写入主库）+ `createConsoleApi(...)`，并通过 `registerMemoryServerCliCommands(memory, { console })` → `startMemoryServer({ console })` 注入。
- daemon 端到端 smoke 测试（console/candidates-smoke.test.ts）验证：enqueue → REST 列出 pending → REST approve → 写入主库 → 不再 pending。
- 既有 overview/lookup/graph/jobs 也随之在 daemon 可达。

**仍缺（observe 自动抽取链路，需后续）**：
- 无 `memory_observe_light` 工具注册和 `AgentFastPathService` 实例（运行中 observation 自动生成 candidate）
- 无 `extract_candidate` job worker（observe → HeuristicTypeExtractor → candidate 入队）

影响：用户/治理可在 Console 审核已有候选并 approve 进主库（端到端可跑）；但"运行中 observation 自动进入候选区"这一自动抽取入口仍需接 observe 工具 + job worker。

### 6.2 provider scope filter spike（A0，为 v0.2 探路）

LanceDB metadata/scope filter 性能 spike 未执行。scope filter 逻辑已在服务层（DatabaseProvider 接口），provider 可替换，spike 只为选定 v0.2 默认 provider。

### 6.3 覆盖率 branch 69.05%（略低于 70%）

statements/lines 已超 80%。branch 缺口主要在 console/、tree/、server/ 等未被 v0.1 核心流程覆盖的分支。需针对性补测。

### 6.4 v0.2 范围（明确不在 v0.1）

跨 appId 复用（Owner/Working Context Scope 分离，5 大改动）、完整 Project Workspace（多 source root + scanner 改造）、cross-product 黄金集、行为监控 pipeline。
