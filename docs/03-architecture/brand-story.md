## 为什么叫梦枢

"梦枢"两个字，拆开看：

**梦** —— 记忆的共享与连续。庄周梦蝶讲的是"主体与记忆的边界"；盗梦空间讲的是"多人进入同一段梦境"。我们用"梦"来指代用户的工作记忆——它不属于某一个 Agent 产品，而是用户自己的、持续存在的、可以被多个 Agent 共同进入的上下文。

**枢** —— 结构与中枢。枢是门轴，是调度的中心。所有 Agent 通过同一个中枢获取用户的工作上下文，不直接触碰底层记忆数据。5 slot 结构化注入是这个中枢的核心输出。

合在一起：**梦枢 = 一个让多个 Agent 共享同一段用户工作记忆的结构化中枢。**

---

## 产品核心能力

梦枢解决一个具体问题：

> 用户在不同 Agent 产品、不同任务之间切换时，工作偏好、项目背景、历史经验和资源不丢失，且能被新 Agent 低成本使用。

三个核心能力：

| 能力 | 说明 |
|------|------|
| **记忆共享** | 同一用户授权下，跨应用复用工作上下文，不需要每次重新说明背景 |
| **结构化注入** | 5 slot 一次性交付完整工作上下文（profile / task_context / rules / experience / resource），不是裸 search |
| **本地优先** | 数据在用户本地，可控、可追溯、可撤销；不依赖云端 |

---

## 产品理念

1. **用户是记忆的主体**。Agent 是访问者，不是拥有者。记忆不绑定某个产品。
2. **结构化优于检索**。Agent 启动时拿到的是整理好的上下文，不是一堆搜索结果。
3. **越用越懂**。用户的偏好、规则、经验随使用沉淀，跨会话持续生效。
4. **安全可解释**。任何被注入的上下文都能追溯来源，private/revoked/stale 不会误注入。

---

## 5 Slot

梦枢的核心输出是 5 个结构化槽位，每个回答工作中的一个问题：

| Slot | 回答 |
|------|------|
| `profile` | 我为谁工作？用户的偏好和协作方式 |
| `task_context` | 我在做什么？当前目标、阶段、约束 |
| `rules` | 不能做什么？长期规则和边界 |
| `experience` | 之前怎么做过？历史经验和反馈 |
| `resource` | 有什么资源？文件、链接、工具、数据源 |

## 与旧名的关系

`memory-autodb` 是早期工程代号。"memory"太泛，"autodb"把产品压成数据库。**梦枢**取代它作为正式产品名，体现跨应用记忆共享和结构化中枢的定位。

具体的命名迁移方案见 [naming-decision.md](./naming-decision.md)。
